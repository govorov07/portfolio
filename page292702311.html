2. SSM Описание

2.1. Введение в проблематику

На данный момент в сервисе AccOrd у пользователей есть ряд концептуальных сложностей с регистрацией данных в ОРД.

Пользователь сейчас не понимает:

  • все ли отправленные сущности были успешно отправлены в ОРД?
  • Все ли сущности были успешно зарегистрированы в ЕРИР?
  • Корректно ли подана отчетности?

На данный момент пользовательский путь по регистрации данных следующий:

  1. Выбирает данные для отправки
  2. Производите отправку
    1. Система формирует flow_input
    2. Система создает заявку/отчет (ord_request)
  3. Пользователь после регистрации переходит в раздел заявки/отчеты
  4. Скачивает Excel за месяц  (иной возможности проверить факт полной и корректной отправки отсутствует)
  5. Изучает Excel на предмет факт полной успешной регистрации данных (подачи отчетности, чаще всего)

2.2. Корневая суть проблемы

Проблема, с точки зрения пользователя, заключается в том, что система не предлагает пользователю прозрачный способ понять и изучить итоги проведенной регистрации сущностей, а также не предоставляет инструменты для проведения данных изысканий.

2.3. Концептуальная модель

Необходимо:

Необходимо произвести ряд доработок в функционале, который напрямую или косвенно связан с регистрацией данных в ОРД, а именно:

  • Статусы заявок/отчетов (результат работы flow по регистрации)
  • Сам процесс отправки
  • Работа с Excel-выгрузками по заявкам/отчетам

Предлагаемое решение:

  2. Система формирует слепок (заявку/отчет)
  3. Система производит отправку данных в ОРД
  4. Пользователь изучает итоги регистрации

3Excel-выгрузки в разрезе одной заявки/отчета

На данный момент пользователь обладает возможностью произвести скатывание Excel-выгрузки только в разрезе целого месяца.

Проблема такой реализации в том, что в пользователи могут отправить одну и ту же сущность в течение одного месяца несколько раз, как следствие в Excel-выгрузке будет одна и та же сущность встречаться несколько раз вне зависимости от своего состояния (иначе "версии")

Предлагается:

Позвоилить пользователю работать с заявками и отчетами не только в рамках целого месяца, но и в рамках отдельно взятой заявки/отчета.

Данная доработка улучшит пользовательский опыт использования сервиса, сократит пользовательский путь при изучении итогов отправки, а также позволит изучить итоги проведенной регистрации данных в рамках отправленного пакета конкретным пользователем

Концепция реализации:

Пересмотр концепции отправки данных в ОРД

Описание

Предлагается реализовать концептуально следующие процессы, с точки зрения пользователи:

  1. Формирование заявки/отчета (формирование слепка в исторических моделях)
  2. Отправка в ОРД заявки/отчета (отправка слепка)

Техническое описание

На данный момент текущую реализацию можно описать следующим образом:

  1. Пользователь отправляет данные (нажимает кнопку на фронте)
  2. FE вызывает метод:
    1. API POST /createive_ord_reg - если регистрируется креатив
    2. API POST /stat_ord_reg - если подается отчетность (акты/статистика)
  3. BE принимает запрос
  4. BE обращается к методу (для простоты назовем его collet_flow_input_method)
    1. BE собирает дерево сущностей, формирует словарь с UUID`ами сущностей для отправки
    2. BE создается запись в таблице FLOW, чтобы в дальнейшем EXECUTOR начал ее обрабатывать 
    3. BE создает заявку/отчета (запись в таблице ord_request)
  5. Работа EXECUTOR`а:
    1. Берет запись из таблицы FLOW
    2. Обращается к словарю flow_input
    3. BE обаращется к методу, согласно котормоу создает слепок в исторических таблицах с необходимым деревом сущностей (для простоты назовем метод create_history_method)
    4. Производит отправку данных

Предлагается:

Пересмотреть два описанных выше метода:

  • collet_flow_input_method 
  • create_history_method

Изменить их так, чтобы UX был следующим:

  1. Пользователь отправляет данные (нажимает кнопку на фронте)
  2. FE вызывает метод:

  4. BE обращается к методу (для простоты назовем его collet_flow_input_method)
    1. BE собирает дерево сущностей, формирует словарь с UUID`ами сущностей для отправки
    2. BE создает заявку/отчета (запись в таблице ord_request)
    3. BE обаращется к методу, согласно котормоу создает слепок в исторических таблицах с необходимым деревом сущностей (для простоты назовем метод create_history_method)
  5. Далее пользователь через интерфейс обращается к сформированной заявке/отчету
  6. Изучает данные для отправки
  7. Производит отправку:
    1. {API POST метод для регистрации креатива}
    2. {API POST метод для подачи отчетности (акт/статистика)}
  8. BE принимает запрос
  9. BE обращается к методу {название нового метода}
    1. BE создается запись в таблице FLOW, чтобы в дальнейшем EXECUTOR начал ее обрабатывать 
  10. Работа EXECUTOR`а:
    1. Берет запись из таблицы FLOW
    2. Обращается к словарю flow_input
    3. BE обаращется к методу, согласно котормоу создает слепок в исторических таблицах с необходимым деревом сущностей (для простоты назовем метод create_history_method)
    4. Производит отправку данных

Статусная модель заявок/отчетов или счетчики

Предлагается реализовать статусную модель для заявок и отчетов (запись ord_request)

Статусная модель предполагается следующая:


ЦветОписаниеКомментарий
1СерыйТребует отправкиКогда пользователь ее только создал и еще не отправил, так как изучает
2СинийВ работеКогда процесс регистрации уже начался, но еще не закончился
3ЗеленыйПринято в ЕРИРКогда все сущности по итогу регистрации (по окончании работы FLOW) в статусе "Принято в ЕРИР" (ERIR_SUCCESS). Никаких ошибок нет
4КрасныйОшибкаКогда хотя бы одна сущность по итогу регистрации (по окончании работы FLOW) в статусе "Ошибка ОРД/ЕРИР" (ORD_ERROR, ERIR_ERROR) - бизнесовая ошибка; техническая ошибка (ORD_FAILED).

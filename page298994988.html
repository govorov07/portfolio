2. SSM Описание

2.1. Введение в проблематику

Имеется проблема наличие rps_core_schema – 1. [SSM] Черновик и Чистовик

В данной статье предлагается рассмотреть план реализации доработок по "Упразднение rps_core_schema. Часть 1: API методы для UI". Поскольку API методы вызываются на интерфейсе, какие-либо доработки напрямую могут повлиять на пользователя.

Но раз это упразднение схемы в базе данных, зачем тут нужны доработки на стороне FE (UI) ?

Дело в том, что на данный момент реализация обычного просмотра и просмотра в режиме регистрации очень сильно отличается, как на FE, так и на BE.

Список проблем

ПроблемаНа кого влияетОписание влияния
API методы возвращают разные тела ответовBEОтсутствие стандартизации методов
Сложность реализации, поддержки
FEРазный FLOW для вызова методов

Сложности в поддержании кодовой базы:

  • Увеличение возможности возникновения багов
  • Трудозатраты на изменения (даже по БФТ)
Отсутствие стандартизации и унификации при реализации сервисов, компонентов
API методы при просмотре в режиме регистрации отдают сложный DTO (уровень вложенности до 5)

FE

Сложность реализации, поддержки

BE

Сложность реализации, поддержки


Эта задача , концептуально, под собой подразумевает следующие риски и проблемы:

  • Переработка текущих решений – могут быть баги (критичные)
  • Нагрузка на команды:
    • Аналитика
    • Разработка
    • Тестирование
  • Согласование таких доработок

2.2. Корневая суть проблемы

Отсутствие унификации реализации API методов, что приводят к:

  • Сложностям при поддержке и доработках
  • Большое кол-во багов

1Только аналитика, без какой-либо разработкиОписание без разработки. Функционал ждет своей реализации

  • На FE и BE различается реализация обычного просмотра и просмотра в режиме регистрации, что ведет к сложности поддержания кодовой базы, возникновению неочевидных багов, отсутствию стандартизации:

    • Разный FLOW по обращению к методам, 
    • Отсутствие стандартизации и унификации
3Версионирование

Реализовать везде, но с учетом версионирования, как на BE, так и на FE

  1. Провести аналитику
  2. Описать
  3. Реализовать на BE
  4. Реализовать на FE
  • Безопасное решение
  • Большая нагрузка на FE, BE
  • Нагрузка на QA
4Постепенная реализацииВ случае если нужны доработки по БФТ какого-то метода. В рамках доработок учесть еще и переработку API
  1. Провести аналитику по БФТ
  2. Описать доработку соотв метода с учетом его перевода на rps_draft_schema
  3. Реализовать на BE
  4. Реализовать на FE
  5. Планово протестировать в рамках БФТ
  • Безопасное решение
  • QA нагружается более планово
  • Очень долгая реализация в полном объеме. Данная задача сильно растянется во времени, что задержит сильно упразднение rps_core_schema
5Менее болезненные переработки

Реализовать методы v2 на чтение из core_schema (по сути копия текущих методов) таким образом, чтобы тело ответа не изменилось. То есть условно просто заменить в sql-запросе одну схему и таблицу на другие (соответствуюбщие из rps_draft_schema).

А уже затем в будущем отдельно заняться унификацией методов (можно по принципу №4 из таблицы)

  1. Провести аналитику
  2. Описать
  3. Реализовать на BE
  • Безопасное решение
  • QA нагружается не так сильно
  • FE толком не дорабатывается. Надо только прописать другой эндпоинт
  • На FE все еще остаются все имеющиеся сложности:
    • Разный FLOW по обращению к методам, 
    • Разные тела ответов
    • Отсутствие стандартизации и унификации реализации
    • и тд

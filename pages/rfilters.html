Сводка проекта

Задача

Провести редизайн раздела "События" с целью повышения удобства использования, скорости доступа к информации и эффективности анализа истории работы умного дома.

Цель

Создать интуитивно понятный, информативный и гибкий интерфейс для просмотра и фильтрации событий, что повысит удовлетворенность пользователей и ценность продукта.

Обоснование

Текущий интерфейс раздела "События" не справляется с растущим объемом и разнообразием данных. Пользователям сложно:

  • Быстро найти нужное событие в длинном, однородном списке.
  • Понять контекст и источник события без дополнительных действий.
  • Анализировать закономерности (например, срабатывание датчика движения в определенное время).
  • Отфильтровать события по критически важным параметрам, таким как устройство или тип события.

Это приводит к потере времени, разочарованию и неэффективному использованию возможностей умного дома.

Целевая аудитория

  • Основные пользователи: Владельцы и постоянные жители умного дома.
  • Второстепенные пользователи: Специалисты поддержки.

Что нужно сделать

Бизнес-цели

  • BG1: Повысить удобство использования и удовлетворенность пользователей продуктом "Умный дом".
  • BG2: Увеличить вовлеченность пользователей в работу с историей событий.
  • BG3: Снизить нагрузку на службу поддержки.
  • BG4: Заложить масштабируемую основу интерфейса для добавления новых типов событий в будущем.

Бизнес-требования

  • BR1: Обеспечить четкое и структурированное визуальное отображение каждого события в списке.
  • BR2: Предоставить пользователю инструменты для гибкой фильтрации списка событий по ключевым параметрам.
  • BR3: Реализовать эффективный и интуитивно понятный механизм просмотра событий за разные периоды времени.
  • BR4: Обеспечить возможность быстрого понимания статуса и важности события.

Функциональные требования

Отображение списка событий

  • FR1.1: Список должен отображаться в хронологическом порядке (последние сверху).
  • FR1.2: Каждое событие в списке должно содержать:
    • Четкая, понятная иконка, визуально обозначающая тип события (тревога, информация, предупреждение и т.д.).
    • Время и дата события в удобочитаемом формате (например, "Сегодня, 14:30" или "23 мая, 09:15").
    • Краткий заголовок события (e.g., "Датчик движения активирован", "Дверь открыта").
    • Описание с деталями (e.g., "Кухня", "Камера у входа").
    • Название устройства-источника события.
  • FR1.3: Реализовать группировку событий по датам (например, "Сегодня", "Вчера", "23 мая 2024").
  • FR1.4: Разные типы событий должны иметь визуальное различие (цветом или иконкой), например:
    • События безопасности/тревоги — акцентный цвет (например, красный/оранжевый).
    • Информационные события — нейтральный цвет (например, синий/серый).
    • Предупреждения — желтый/янтарный.
    • Финансовые события (баланс) — зеленый.

Система фильтров

  • FR2.1: Реализовать панель фильтров с возможностью применения одного или нескольких условий одновременно.
  • FR2.2: Фильтры должны включать:
    • Фильтр по периоду: Выбор даты начала и даты окончания из календаря.
    • Фильтр по времени суток: Опциональный выбор "Утро", "День", "Вечер", "Ночь".
    • Фильтр по группам событий: Выбор из предустановленных групп ("Безопасность", "Устройства", "Камеры", "Финансы").
    • Фильтр по типам событий: Множественный выбор конкретных типов внутри групп (e.g., "Движение", "Открытие двери", "Зарядка устройства", "Обнаружено лицо").
    • Фильтр по устройствам: Множественный выбор конкретных устройств из списка (e.g., "Датчик движения в гостиной", "Камера заднего двора", "Умная розетка Кофеварка").
  • FR2.3: Предусмотреть кнопку "Сбросить фильтры" для быстрого возврата к полному списку событий.
  • FR2.4: На панели фильтров должно отображаться количество активных фильтров.

Производительность и данные

  • FR3.1: Обеспечить быструю загрузку списка событий (целевое время - менее 2 секунд для первых 50 записей).
  • FR3.2: Реализовать постраничную загрузку или бесконечный скролл для работы с большими объемами истории.
  • FR3.3: Фильтрация и поиск должны выполняться без перезагрузки всей страницы (использование AJAX/API).

Нефункциональные требования

  • NFR1: Usability (Удобство использования): Интерфейс должен быть интуитивно понятен для пользователя с базовыми техническими навыками. Все элементы управления должны соответствовать гайдлайнам платформы (iOS/Android/Material Design).
  • NFR2: Performance (Производительность): Взаимодействие с интерфейсом (скролл, применение фильтров) должно быть плавным, без заметных лагов.
  • NFR3: Consistency (Согласованность): Новый дизайн должен быть визуально и концептуально согласован с остальными разделами приложения "Умный дом".

Ограничения

  • Дизайн должен быть адаптирован под мобильные устройства (смартфоны) и, опционально, под веб-интерфейс.
  • Реализация не должна требовать изменения архитектуры backend-системы для генерации событий.

Методы сбора данных и валидации

  1. Аналитика продукта: Внедрение событий (events) в приложение: Screen_View_Events, Filter_Applied, Event_Search_Performed, Time_To_Find_Event.
  2. Опросы пользователей: Проведение кратких in-app опросов через 1-2 недели после релиза: "Стало ли вам проще находить нужные события?" (Да/Нет/Затрудняюсь ответить).
  3. Данные поддержки: Мониторинг тикетов в системе поддержки (Zendesk, Jira Service Desk) по соответствующим категориям.
  4. A/B тестирование (опционально, но желательно): Выпуск новой версии не для 100% аудитории, а для 50% (группа А), оставив старую версию для группы B. Сравнение всех вышеуказанных метрик между группами предоставит чистейшие данные об эффективности фичи.

Пользовательские сценарии

Job Stories

Job Stories фокусируются на ситуации, мотивации и цели, что помогает разрабатывать более гибкие и интуитивные решения.

JS1: Просмотр и поиск событий

  • Когда я слышу странный звук срабатывания сигнализации, пока меня нет дома,
  • Я хочу быстро найти соответствующее событие за последние 10 минут и понять, что именно сработало,
  • Чтобы немедленно оценить серьезность ситуации и принять решение (позвонить соседу, проигнорировать ложное срабатывание и т.д.).

JS2: Анализ и выявление закономерностей

  • Когда я пытаюсь понять, почему растет потребление электроэнергии,
  • Я хочу увидеть все события включения умных розеток и термостатов за последнюю неделю, сгруппированные по устройствам,
  • Чтобы выявить устройство, которое работает чаще или дольше обычного, и настроить его расписание.

JS3: Фильтрация для диагностики проблемы

  • Когда датчик двери гаража начал показывать "низкий заряд батареи",
  • Я хочу отфильтровать все события, связанные только с этим датчиком, за последний месяц,
  • Чтобы понять, как давно началась проблема и как быстро садится батарея, и принять решение о ее замене.

JS4: Проверка финансовых операций

  • Когда я вижу списание денег за подписку на сервис "Умный дом",
  • Я хочу легко найти событие об изменении баланса в этот день и сопоставить его с другими финансовыми операциями,
  • Чтобы убедиться в корректности списания и отсутствии ошибок.

JS5: Быстрая оценка общей обстановки

  • Когда я просыпаюсь утром,
  • Я хочу бегло просмотреть все важные события, которые произошли за ночь (отсеяв информационные уведомления),
  • Чтобы быть в курсе, не было ли несанкционированных проникновений или срабатываний тревоги, пока я спал.

User Stories

User Stories следуют классическому формату "Как [роль], я хочу [функцию], чтобы [выгода]". Они более тесно связаны с конкретными функциональными требованиями.

Epic: Улучшенное отображение списка событий

  • US1:
    • Как пользователь,
    • Я хочу, чтобы события в списке были сгруппированы по датам (Сегодня, Вчера, Ранее),
    • Чтобы мне было проще ориентироваться во времени произошедших событий.
  • US2:
    • Как пользователь,
    • Я хочу видеть четкую иконку и цветовое кодирование для каждого типа события (тревога, предупреждение, информация),
    • Чтобы я мог мгновенно оценить важность события, не читая текст.
  • US3:
    • Как пользователь,
    • Я хочу видеть в каждом событии заголовок, описание, время и устройство-источник,
    • Чтобы получить полную информацию о событии без необходимости переходить в детальный просмотр.

Epic: Гибкая система фильтрации

  • US4:
    • Как пользователь,
    • Я хочу иметь возможность фильтровать события по дате и времени (с/по, время суток),
    • Чтобы быстро сузить круг поиска до нужного временного промежутка.
  • US5:
    • Как пользователь,
    • Я хочу выбирать для фильтрации конкретные устройства или группы устройств (например, все датчики на первом этаже),
    • Чтобы видеть события только от интересующих меня устройств.
  • US6:
    • Как пользователь,
    • Я хочу фильтровать события по их типу (движение, открытие, низкий заряд) и группе (Безопасность, Устройства, Финансы),
    • Чтобы найти события схожей природы, даже если они пришли с разных устройств.
  • US7:
    • Как пользователь,
    • Я хочу применять несколько фильтров одновременно (например, "Датчик движения на кухне" + "Сегодня" + "Вечер"),
    • Чтобы максимально точно найти именно то событие, которое мне нужно.
  • US8:
    • Как пользователь,
    • Я хочу видеть количество активных фильтров и иметь возможность быстро сбросить их все одной кнопкой,
    • Чтобы легко управлять фильтрацией и возвращаться к полному списку событий.

Epic: Производительность и удобство использования

  • US9:
    • Как пользователь с медленным интернетом,
    • Я хочу, чтобы список событий загружался быстро и плавно,
    • Чтобы не ждать и не испытывать раздражение при работе с историей.
  • US10:
    • Как пользователь, изучающий историю,
    • Я хочу, чтобы при прокрутке списка подгружались новые события (бесконечный скролл),
    • Чтобы беспрепятственно просматривать длинную историю без переключения страниц.

Организационные задачи/Смежники

Указать с какими подразделениями надо вести разработку совместно, чтобы проект сложился и перечень задач, которые должны быть выполнены для того, чтобы функциональностью можно было пользоваться в проме. Обязательно указывать ответственных лиц.

Definition of done (DoD)

DoD для одной User Story

Каждая User Story (например, US1, US2, US4 и т.д.) считается «Выполненной», когда:

  1. Код написан и отрефакторен:
    • Код соответствует стандартам и гайдлайнам проекта (code style).
    • Проведен рефакторинг, код чистый и поддерживаемый.
  2. Функциональность реализована в полном объеме:
    • Реализация соответствует описанию в User Story и принятым к выполнению задачам.
    • Все сценарии использования, описанные в истории, работают корректно.
  3. Проведено модульное и интеграционное тестирование (разработчиком):
    • Написаны и успешно пройдены юнит-тесты для нового кода.
    • Критическая логика (например, работа фильтров) покрыта интеграционными тестами.
  4. UI соответствует макетам:
    • Внешний вид и поведение элементов интерфейса в точности соответствуют утвержденным дизайн-макетам (Figma, Sketch) для всех поддерживаемых разрешений (мобильное, планшетное).
  5. Кросс-браузерная и кроссплатформенная совместимость:
    • Функциональность и внешний вид проверены и работают корректно на:
      • Мобильные ОС: Последние версии iOS и Android.
      • Браузеры (для веба): Chrome, Firefox, Safari, Edge (последние 2 версии).
  6. Проведено ручное тестирование QA:
    • Тестировщик проверил функциональность по тест-кейсам.
    • Все найденные баги исправлены и перепроверены.
    • Тестировщик поставил статус "Passed" на задачу.
  7. Нет регрессии:
    • Существующий функционал приложения, не связанный напрямую с фичей, продолжает работать корректно (проверяется прогоном регрессионных тестов).
  8. Код проверен (Code Review):
    • Код проверен как минимум одним другим разработчиком.
    • Все замечания по код-ревью устранены.
  9. Документация обновлена (при необходимости):
    • Обновлена внутренняя техническая документация, если были изменены API.
    • Внесены правки в документацию для пользователей (например, раздел справки в приложении), если фича этого требует.

DoD для всей Фичи

Вся Фича считается «Выполненным» и готовым к выпуску в продакшен, когда:

  1. Все User Stories в рамках эпика выполнены: Каждая отдельная история удовлетворяет всем критериям DoD, указанным выше.
  2. Проведено всестороннее тестирование QA:
    • Функциональное тестирование: Все фильтры, отображение, группировка, производительность работают как единое целое.
    • Приемочное тестирование (UAT): Фича проверена продакт-менеджером/стейкхолдером и подтверждена, что она решает заявленные бизнес-задачи.
    • Нагрузочное тестирование (опционально, но желательно): Проверена работа интерфейса с большим объемом событий (тысячи записей) — интерфейс не "подвисает".
    • Тестирование удобства использования (Usability Testing): Проведено тестирование на фокус-группе или внутренними экспертами, подтвердившее улучшение UX.
  3. Аналитика и мониторинг настроены:
    • В код внедрены все необходимые события для отслеживания ключевых метрик (просмотры раздела, применение фильтров, время поиска).
    • Настроены дашборды в системах аналитики (Amplitude, Mixpanel) для мониторинга метрик из раздела "Критерии успеха".
  4. Поддержка уведомлена:
    • Служба поддержки проинформирована о новом функционале, им предоставлена документация и ответы на возможные частые вопросы (FAQ).
  5. Деплой в продакшен:
    • Код задеплоен в продовольственное окружение и прошел пост-деплойную проверку.
    • Отсутствуют критические ошибки в мониторинге (Sentry, Crashlytics и т.д.) в течение первых 24 часов после релиза.
  6. Критерии успеха начали отслеживаться:
    • Запущены процессы сбора данных для расчета метрик успеха (увеличение использования раздела, снижение времени поиска и т.д.).

Пример применения DoD к конкретной User Story

User Story US5: "Как пользователь, я хочу выбирать для фильтрации конкретные устройства, чтобы видеть события только от интересующих меня устройств."

DoD для US5:

  • Код виджета/меню выбора устройств написан.
  • Реализован бэкенд-API для получения списка устройств пользователя.
  • Написаны юнит-тесты для функции фильтрации по устройствам.
  • UI элемента выбора устройств соответствует макетам из Figma.
  • Функция проверена на iOS и Android (выбор, применение, сброс).
  • QA проверил: можно выбрать одно устройство, несколько; фильтрация применяется корректно; список обновляется без перезагрузки страницы.
  • Проверено, что старые фильтры по дате не сломались при добавлении нового.
  • Код проверен другим разработчиком.
  • Событие filter_by_device_applied отправляется в аналитику при использовании фильтра.
